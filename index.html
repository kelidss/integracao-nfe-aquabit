Documentação Técnica: Integração de Nota Fiscal Eletrônica (NFe) no Sistema Aquabit

Versão: 2.0 | Data: 30 de Setembro de 2025

1. Visão Geral

Este documento detalha a arquitetura e o plano de implementação para a integração de sistemas de emissão de Nota Fiscal Eletrônica (NFe) na plataforma Aquabit. O objetivo é automatizar e gerenciar o ciclo de vida das notas fiscais geradas a partir de movimentações financeiras registradas no sistema.

1.1. Objetivos Principais

  • Automatizar a Emissão: Gerar NFes automaticamente para todas as movimentações financeiras concluídas na plataforma.
  • Integração com Revgás: Integração dedicada com o provedor Revgás para atender as necessidades específicas do setor de combustíveis.
  • Gestão Completa: Permitir a consulta de status, listagem e o download do Documento Auxiliar da Nota Fiscal Eletrônica (DANFE) e do arquivo XML.
  • Rastreabilidade: Manter um histórico detalhado e logs completos de todas as transações fiscais para auditoria e controle.

1.2. Tabelas Base de Dados

A integração utilizará as seguintes tabelas do banco de dados como fonte de informação para a geração das notas fiscais:

Tabela Descrição
core_movimentacao Armazena as movimentações financeiras - fonte principal para emissão de NFe.
core_movimentacaoitem Detalha os itens específicos de cada movimentação com quantidades e valores.
core_categoria Define as categorias de produtos/combustíveis - substitui a tabela de produtos.
core_cliente Dados dos clientes necessários para emissão da NFe (CNPJ/CPF, endereço, etc.).

1.3. Análise dos Requisitos Mínimos para Emissão de NFe

Para garantir a conformidade fiscal e a correta emissão das NFes, é necessário analisar as tabelas existentes e identificar os campos obrigatórios. Abaixo estão os requisitos mínimos organizados por categoria:

1.3.1. Dados do Emitente (Obrigatórios)

Estes dados devem estar configurados na propriedade/empresa (p.ex. tabela core_perfilpropriedade):

Nota: Recomendamos armazenar os dados do emitente na tabela core_perfilpropriedade ou equivalente, que contenha CNPJ, inscrição estadual, razão social, endereço e regime tributário. Isso facilita a reutilização para múltiplas propriedades/postos.

Informações necessárias para compor a NFe:

  • Data da Operação: Data e hora da movimentação
  • Valor Total: Valor total da operação
  • Quantidade: Quantidade de cada item (em core_movimentacaoitem)
  • Valor Unitário: Preço unitário de cada item
  • Valor Total do Item: Quantidade × Valor Unitário

1.3.5. Campos Recomendados para Implementação

Para uma integração completa com a Revgás, sugerimos a criação/validação dos seguintes campos:

  1. Na tabela cliente: Validar se todos os campos de endereço estão presentes e obrigatórios
  2. Na tabela categoria: Adicionar campos específicos para tributação (NCM, CFOP, CST, etc.)
  3. Na tabela propriedade: Garantir que dados fiscais da empresa estão completos
  4. Configuração de série e numeração: Sistema para controle sequencial das NFes

2. Arquitetura da Solução

A solução foi desenhada de forma modular, utilizando uma arquitetura baseada em Models, Services e Adapters para garantir escalabilidade e fácil manutenção. A integração inicial foca exclusivamente no provedor Revgás, especializado no setor de combustíveis.

2.1. Estrutura de Models

Os models abaixo representam a estrutura de dados no banco de dados para gerenciar provedores, configurações, notas fiscais e logs.

# apps/integracao/models.py

class ProvedorNFe(BaseClass):
    """
    Representa os provedores de NFe disponíveis no sistema.
    """
    nome = models.CharField(max_length=100)
    slug = models.SlugField(unique=True)
    api_url = models.URLField()
    ativo = models.BooleanField(default=True)


class ConfiguracaoNFe(BaseClass):
    """
    Armazena as configurações de NFe para cada propriedade (cliente),
    incluindo credenciais e ambiente de operação.
    """
    propriedade = models.ForeignKey('core.Propriedade', on_delete=models.CASCADE)
    provedor = models.ForeignKey(ProvedorNFe, on_delete=models.CASCADE)
    api_token = models.CharField(max_length=255)
    ambiente = models.CharField(max_length=20)  # 'homologacao' ou 'producao'
    certificado_digital = models.TextField(blank=True)
    senha_certificado = models.CharField(max_length=100, blank=True)


class NotaFiscal(BaseClass):
    """
    Registra cada NFe emitida, seu status, chave de acesso e
    armazena os arquivos XML e DANFE. Também persiste o retorno bruto
    da API do provedor (JSON/text) e o código de status HTTP.
    """
    movimentacao = models.ForeignKey('core.Movimentacao', on_delete=models.CASCADE)
    provedor = models.ForeignKey(ProvedorNFe, on_delete=models.CASCADE)
    numero_nota = models.CharField(max_length=50, blank=True)
    serie = models.CharField(max_length=10, blank=True)
    chave_acesso = models.CharField(max_length=44, blank=True)
    status = models.CharField(max_length=50)
    data_emissao = models.DateTimeField(null=True, blank=True)
    xml_content = models.TextField(blank=True)
    pdf_danfe = models.FileField(upload_to='nfe/danfe/', blank=True)
    protocolo_autorizacao = models.CharField(max_length=100, blank=True)
    provedor_response = models.TextField(blank=True)  # JSON/text com retorno do provedor
    provedor_status_code = models.IntegerField(null=True, blank=True)


class LogNFe(BaseClass):
    """
    Mantém um log detalhado de todas as operações relacionadas a uma NFe
    para fins de depuração e auditoria. Cada entrada deve guardar request
    e response quando aplicável.
    """
    nota_fiscal = models.ForeignKey(NotaFiscal, on_delete=models.CASCADE, related_name='logs')
    operacao = models.CharField(max_length=50)  # 'emissao', 'consulta', 'cancelamento', 'webhook'
    request_payload = models.TextField(blank=True)
    response_payload = models.TextField(blank=True)
    status_code = models.IntegerField(null=True, blank=True)
    mensagem = models.TextField(blank=True)
    criado_em = models.DateTimeField(auto_now_add=True)

2.2. Camada de Serviço (Services)

O NFEService atua como a camada central de lógica de negócios, orquestrando as operações de NFe e utilizando um padrão de Factory para selecionar o Adapter correto para cada provedor.

# apps/integracao/services/nfe_service.py

class NFEService:
    """
    Serviço base para orquestrar as operações de NFe.
    """

    def __init__(self, configuracao: ConfiguracaoNFe):
        self.configuracao = configuracao
        self.adapter = self._get_adapter()

    def _get_adapter(self):
        """
        Factory Method que instancia o adapter correto com base
        no provedor configurado. Atualmente suporta apenas Revgás.
        """
        adapters = {
            'revgas': RevgasAdapter,
        }
        adapter_class = adapters.get(self.configuracao.provedor.slug)
        if not adapter_class:
            raise ValueError("Provedor de NFe não suportado. Apenas Revgás é suportado atualmente.")
        return adapter_class(self.configuracao)

    def emitir_nfe(self, movimentacao: Movimentacao) -> NotaFiscal:
        """
        Coordena o processo de emissão de uma NFe para uma movimentação específica.
        """
        # ...lógica de negócio...
        return self.adapter.emitir(movimentacao)

    def consultar_nfe(self, nota_fiscal: NotaFiscal) -> dict:
        """
        Consulta o status atual de uma NFe já emitida.
        """
        # ...lógica de negócio...
        return self.adapter.consultar(nota_fiscal)

    def baixar_danfe(self, nota_fiscal: NotaFiscal) -> bytes:
        """
        Realiza o download do PDF da DANFE.
        """
        # ...lógica de negócio...
        return self.adapter.baixar_pdf(nota_fiscal)

    def baixar_xml(self, nota_fiscal: NotaFiscal) -> bytes:
        """
        Realiza o download do arquivo XML da NFe.
        """
        # ...lógica de negócio...
        return self.adapter.baixar_xml(nota_fiscal)

2.3.4. Tabelas sugeridas: nfe e nfe_logs (persistência de retornos da provedora)

Abaixo uma DDL proposta, pensada para PostgreSQL. Observações principais: o campo provider_return é do tipo JSONB para facilitar consultas/filtragem por chaves específicas; a coluna emitente_id referencia core_perfilpropriedade(id) conforme alinhado.

-- Tabela principal de NFe
CREATE TABLE IF NOT EXISTS nfe (
    id BIGSERIAL PRIMARY KEY,
    emitente_id BIGINT REFERENCES core_perfilpropriedade(id) ON DELETE SET NULL,
    movimentacao_id BIGINT REFERENCES core_movimentacao(id) ON DELETE SET NULL,
    chave VARCHAR(44) UNIQUE NOT NULL,
    numero VARCHAR(50),
    serie VARCHAR(20),
    modelo VARCHAR(10),
    data_emissao TIMESTAMPTZ,
    total NUMERIC(14,2),
    status VARCHAR(50) NOT NULL DEFAULT 'pending',
    provider VARCHAR(100),
    provider_return JSONB, -- retorno bruto/estruturado da API provedora
    provider_return_received_at TIMESTAMPTZ,
    created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
    updated_at TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE INDEX IF NOT EXISTS idx_nfe_chave ON nfe(chave);
CREATE INDEX IF NOT EXISTS idx_nfe_emitente_id ON nfe(emitente_id);

-- Tabela de logs históricos de interação com provedores (requests/responses/erros)
CREATE TABLE IF NOT EXISTS nfe_logs (
    id BIGSERIAL PRIMARY KEY,
    nfe_id BIGINT NOT NULL REFERENCES nfe(id) ON DELETE CASCADE,
    attempt INTEGER NOT NULL DEFAULT 1,
    direction VARCHAR(10) NOT NULL CHECK (direction IN ('outbound','inbound')),
    endpoint VARCHAR(255),
    status_code INTEGER,
    request JSONB,
    response JSONB,
    error TEXT,
    created_at TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE INDEX IF NOT EXISTS idx_nfe_logs_nfe_id ON nfe_logs(nfe_id);
CREATE INDEX IF NOT EXISTS idx_nfe_logs_created_at ON nfe_logs(created_at);

Com esse desenho, a rota /api/v2/nfe/ deve:

  • Criar/atualizar o registro em nfe (idempotência por chave quando aplicável).
  • Persistir o retorno da API provedora em nfe.provider_return e marcar provider_return_received_at.
  • Inserir entradas em nfe_logs para cada tentativa/response, com direction = 'outbound' para o request e 'inbound' para a resposta do provedor.

2.3.5. Detalhes do XML e retorno do provedor

O comportamento esperado das APIs provedoras geralmente envolve dois artefatos principais que precisamos persistir/trackear:

  1. O XML autorizado da NFe (arquivo padrão da SEFAZ) — contém todos os campos fiscais oficiais e o protocolo de autorização.
  2. O retorno da API do provedor (JSON/text) — resposta da integração com provedores como Revgás, que pode trazer metadados, status e um mirror do XML ou um link para download.
Exemplo simples de XML autorizado (trecho)
<nfeProc versao="4.00" xmlns="http://www.portalfiscal.inf.br/nfe">
            <NFe>
                <infNFe Id="NFe99999999999999999999999999999999999999999999" versao="4.00">
                    <ide><nNF>1234</nNF><serie>1</serie><dhEmi>2025-09-30T12:00:00-03:00</dhEmi></ide>
                    <emit><CNPJ>00000000000000</CNPJ></emit>
                    <dest><CNPJ>11111111111111</CNPJ></dest>
                    <det>...detalhes dos itens...
Exemplo de JSON de retorno do provedor (provider_return)
{
            "status": "autorizada",
            "codigo": 100,
            "mensagem": "Autorizado o uso da NF-e",
            "chave": "99999999999999999999999999999999999999999999",
            "numero": "1234",
            "serie": "1",
            "protocolo": "202509300000000",
            "dhRecbto": "2025-09-30T12:00:10-03:00",
            "xml": "...xml completo base64 ou texto...",
            "events": [
                {"tipo":"autorizacao","codigo":100,"descricao":"Autorizado"}
            ]
        }
Mapeamento e recomendações
  • Salvar o XML autorizado: preferível salvar o XML como arquivo no storage (S3 / storage local) e guardar o caminho em uma coluna tipo xml_path ou em xml_content (TEXT) dependendo do volume. Se for armazenar no DB, use xml_content TEXT ou xml_content BYTEA (com compressão).
  • provider_return (JSONB): persistir o JSON completo em nfe.provider_return para auditoria. Crie um índice GIN se for executar consultas por chaves internas: CREATE INDEX idx_nfe_provider_return_gin ON nfe USING gin (provider_return);
  • Normalizar campos críticos: extraia para colunas separadas (ex.: protocolo_autorizacao, numero, chave, provedor_status_code) para consultas rápidas e filtros.
  • Ao inserir logs (nfe_logs), salve ambos request/response como JSONB e registre status_code e eventual stack em error.
  • Se o provedor retornar o XML inline (como no exemplo), prefira salvá-lo no storage e referenciar; mantenha também uma cópia reduzida/assinada no DB se necessário para recuperação rápida.
  • Mascaramento: remova ou mascare segredos/tokens antes de persistir logs visíveis (ex.: bearer tokens, senhas).
  • Valide tamanho: se provider_return for muito grande, avalie armazenar apenas metadados no JSONB e mover o body completo para arquivos em storage referenciados por URL no JSON.
Consultas e extração (exemplos)
-- Buscar NFe por protocolo dentro do JSONB
        SELECT * FROM nfe WHERE provider_return->> 'protocolo' = '202509300000000';

        -- Índice GIN para buscas em provider_return
        CREATE INDEX IF NOT EXISTS idx_nfe_provider_return_gin ON nfe USING gin (provider_return);

Com essas informações adicionadas, a equipe responsável pela API poderá padronizar os campos retornados pelos adapters (por exemplo: sempre retornar chave, numero, protocolo, xml ou xml_url). Se quiser, eu atualizo as partes do documento que descrevem os adapters (ex.: pseudo-código do Revgás) para refletir esse contrato de retorno padronizado.

2.4.1. POST /api/v2/nfe/ (Criar/Pré-registrar NFe)

Função: criar um registro de NFe a partir de uma movimentação e, opcionalmente, disparar emissão assíncrona.

{
  "id": 987,

Consulta estado atual consultando o provedor e atualizando localmente nota_fiscal.provedor_response.

2.5. Exemplo de Adapter Revgás (pseudo-código / Python)

class RevgasAdapter:
    def __init__(self, configuracao: ConfiguracaoNFe):
        self.base_url = configuracao.provedor.api_url
        self.token = configuracao.api_token

    def emitir(self, movimentacao: Movimentacao) -> NotaFiscal:
        payload = self._build_payload(movimentacao)
        resp = requests.post(f"{self.base_url}/nfe/emissao", json=payload, headers={"Authorization": f"Bearer {self.token}"}, timeout=30)
        nota = NotaFiscal.objects.create(
            movimentacao=movimentacao,
            provedor=configuracao.provedor,
            status='processing',
            provedor_response=resp.text,
            provedor_status_code=resp.status_code
        )
        LogNFe.objects.create(nota_fiscal=nota, operacao='emissao', request_payload=json.dumps(payload), response_payload=resp.text, status_code=resp.status_code)
        if resp.status_code == 200:
            data = resp.json()
            nota.numero_nota = data.get('numero')
            nota.serie = data.get('serie')
            nota.chave_acesso = data.get('chave')
            nota.status = data.get('status')
            nota.save()
        return nota

2.6. Tarefas assíncronas (Celery)

Exemplo de task que processa a emissão assincronamente:

from celery import shared_task

@shared_task(bind=True)
def emitir_nfe_task(self, nota_id: int):
    nota = NotaFiscal.objects.get(pk=nota_id)
    adapter = NFEService(nota.configuracao).adapter
    try:
        result = adapter.emitir(nota.movimentacao)
        # atualizar nota com retorno
    except Exception as exc:
        LogNFe.objects.create(nota_fiscal=nota, operacao='emissao', mensagem=str(exc))
        raise self.retry(exc=exc, countdown=60)

2.7. Logs, auditoria e retenção

  • Persistir respostas brutas do provedor em nota_fiscal.provedor_response permite auditoria e reprocessamento.
  • Logs detalhados em integracao_lognfe com payloads e códigos HTTP permitem investigar problemas.
  • Política de retenção: manter logs de até 1 ano por padrão; exportar para storage frio se necessário.

2.8. Webhooks (evitar polling)

Quando o provedor oferece webhooks, é preferível utilizá-los para evitar tarefas que fazem polling contínuo do status. Abaixo uma especificação mínima para implementar um endpoint webhook seguro e idempotente.

Endpoint sugerido

POST /api/v2/nfe/webhook/

Segurança

  • Validação de assinatura HMAC: o provedor deve enviar um cabeçalho como X-Hub-Signature ou X-Signature com HMAC do payload. Verifique utilizando a chave compartilhada armazenada em ConfiguracaoNFe.
  • Validade de IP (opcional): whitelist de IPs do provedor.
  • Idempotência: valide se o evento já foi processado (por exemplo, checando evento_id no payload ou usando chave + protocolo).

Payload mínimo esperado (exemplo)

{
  "evento_id": "evt_abc123",
  "chave": "99999999999999999999999999999999999999999999",
  "tipo": "autorizacao|cancelamento|denegada",
  "protocolo": "202509300000000",
  "status": "autorizada",
  "mensagem": "Autorizado o uso da NF-e",
  "xml_url": "https://provedor/files/nfe/9999.xml", -- opcional
  "xml_base64": "PD94bWwgdmVyc2lvbj0i..." -- opcional
}

Comportamento do endpoint

  1. Validar assinatura/IP.
  2. Verificar se a NFe identificada por chave existe localmente; se existir, atualizar nfe.provider_return, nfe.status e provider_return_received_at, e gravar um registro em nfe_logs (direction='inbound').
  3. Se não existir (protocolo chegou antes do registro local), criar um registro mínimo em nfe com chave, status e salvar o retorno; marcar uma flag created_via_webhook=true (opcional) para rastrear origem.
  4. Responder rapidamente com 200 OK para o provedor assim que o evento for aceito/valido; se houver erro temporário (ex.: DB indisponível), responder 5xx para que o provedor reenvie.

Idempotência e duplicatas

Use evento_id fornecido pelo provedor quando disponível para garantir que o mesmo evento não seja processado duas vezes. Caso não exista, gere um hash do payload e verifique um índice único em nfe_logs(event_hash) antes de inserir.

2.9. Fail-safe: garantir persistência local mesmo em erro de integração

Para evitar perda de dados caso a integração com o provedor falhe (timeout, erro 5xx, etc.), siga a estratégia abaixo:

DDl atualizada (flags e campos de retry)

-- Extensão da tabela nfe com flags de controle
ALTER TABLE nfe
ADD COLUMN emitted BOOLEAN NOT NULL DEFAULT FALSE, -- indica se a NFe foi criada/autorizada no provedor
ADD COLUMN external_sent BOOLEAN NOT NULL DEFAULT FALSE, -- indica se o payload foi enviado ao provedor
ADD COLUMN external_attempts INTEGER NOT NULL DEFAULT 0, -- número de tentativas
ADD COLUMN last_error TEXT; -- último erro ocorrido ao tentar integrar

-- Índice para buscas de NFes pendentes de envio
CREATE INDEX IF NOT EXISTS idx_nfe_pending_send ON nfe((external_sent = false) AND (external_attempts < 5));

Fluxo seguro (pseudocódigo)

# Pseudocódigo simplificado (ex: endpoint POST /api/v2/nfe/)
def criar_nfe(payload):
    # 1) Persistir localmente antes de qualquer tentativa externa
    nfe = db.insert('nfe', {
        'chave': payload.chave,
        'movimentacao_id': payload.movimentacao_id,
        'status': 'pending',
        'external_sent': False,
        'external_attempts': 0
    })

    # 2) Tentar enviar para provedor de forma assíncrona (task/queue)
    enqueue('emitir_nfe', {'nfe_id': nfe.id})

    return nfe

# Worker/Task emitir_nfe
def emitir_nfe_task(nfe_id):
    nfe = db.get('nfe', nfe_id)
    try:
        # marcar tentativa
        db.update('nfe', nfe_id, {'external_attempts': nfe.external_attempts + 1})

        # enviar para provedor
        resp = provedor.emitir(nfe)

        # salvar retorno e marcar como enviado
        db.update('nfe', nfe_id, {
            'provider_return': resp.json(),
            'provider_return_received_at': now(),
            'external_sent': True,
            'emitted': resp.is_autorizada(),
            'status': resp.status_label(),
            'last_error': None
        })
        db.insert('nfe_logs', {...})
    except TransientError as e:
        db.update('nfe', nfe_id, {'last_error': str(e)})
        # requeue/retry controlado pelo worker (exponential backoff)
        raise
    except Exception as e:
        db.update('nfe', nfe_id, {'last_error': str(e)})
        db.insert('nfe_logs', {...})
        # não perde o registro local; pode expor endpoint de retry manual

# Endpoint de retry administrativo
POST /api/v2/nfe/{id}/retry/ -> enfileira novamente emitir_nfe_task se external_attempts < MAX

Endpoint de retry manual

Um endpoint administrativo pode ser exposto para re-tentar envio de NFes que falharam automaticamente. Regras sugeridas:

  • Somente usuários com permissão podem chamar.
  • Verificar external_attempts e limitar re-execuções (ex.: max 5 tentativas).
  • Registrar no nfe_logs quem disparou o retry e por quê.

Comportamento em caso de falha

  • Não deletar o registro local da NFe: manter para auditoria e reprocessamento.
  • Marcar campos auxiliares (last_error, external_attempts) para monitoramento e alertas.
  • Se o provedor confirmar posteriormente via webhook, reconciliar os dados e limpar flags de retry.

4. Endpoints da API

A gestão da integração será realizada através dos seguintes endpoints na API v2.

4.1. Notas Fiscais (/api/v2/nfe/)

Método Endpoint Descrição
GET / Lista todas as NFes emitidas com filtros.
POST / Cria um registro de Nota Fiscal (pré-registro). Ao processar a emissão, o retorno da API do provedor é persistido em nota_fiscal.provedor_response e registros de interação são gravados na tabela log_nfe.
GET /{id}/ Detalha uma NFe específica.
PUT /{id}/ Atualiza os dados de uma NFe (uso restrito).
DELETE /{id}/ Remove um registro de NFe (cancelamento lógico).
POST /{id}/emitir/ Dispara o processo de emissão da NFe.
GET /{id}/consultar/ Consulta o status mais recente da NFe no provedor.
GET /{id}/baixar_danfe/ Realiza o download do arquivo PDF da DANFE.
GET /{id}/baixar_xml/ Realiza o download do arquivo XML autorizado.

4.2. Configurações (/api/v2/configuracao-nfe/)

Método Endpoint Descrição
GET / Lista todas as configurações de NFe.
POST / Cria uma nova configuração para uma propriedade.
GET /{id}/ Detalha uma configuração específica.
PUT /{id}/ Atualiza uma configuração existente.
DELETE /{id}/ Remove uma configuração.

5.1. Variáveis de Ambiente

Variável Descrição Exemplo
NFE_DEFAULT_PROVIDER Provedor padrão a ser utilizado. revgas

5.2. Configuração de Filas (Celery)

Para garantir que a emissão de notas fiscais não impacte a performance da aplicação principal, as operações serão processadas de forma assíncrona utilizando Celery. As tarefas serão roteadas para uma fila dedicada chamada nfe.
